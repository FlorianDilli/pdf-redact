# PDF Redaction App — Implementation Plan

## Goal
Build a single-file, fully client-side web app (`index.html`) that lets users upload searchable PDFs, preview every match of user-specified terms, and export a permanently redacted PDF. Zero server contact. Deployable to GitHub Pages.

---

## Architecture

| Concern | Decision |
|---|---|
| Format | Single `index.html` — all HTML, CSS, and JS inline. No build step. |
| Hosting | Static files only — works on GitHub Pages out of the box |
| Dependencies | `pdf.js` v4 + `pdf-lib` — both loaded from CDN in the HTML |
| State | In-memory JavaScript objects; page is lost on refresh (acceptable for a tool) |
| Privacy | No `fetch()`, no `<img src>` to external URLs, no analytics, no CDN for user data |

---

## Library Choices

### pdf.js (`pdfjs-dist`)
- **Why**: Industry-standard for in-browser PDF rendering and text extraction. Gives access to the full text layer with per-character or per-span bounding boxes via `page.getTextContent()`.
- **Version**: v4.x (latest stable)
- **CDN**: `https://cdnjs.cloudflare.com/ajax/libs/pdf.js/4.x/pdf.min.mjs` (ESM build, no worker file needed)
- **Use for**: Rendering pages to `<canvas>`; extracting text content with position data.

### pdf-lib
- **Why**: Pure-JS PDF generation library. Can create new PDFs, embed images as pages, and set page dimensions. No Node.js dependency.
- **Version**: latest stable
- **CDN**: `https://cdnjs.cloudflare.com/ajax/libs/pdf-lib/1.17.x/pdf-lib.min.js`
- **Use for**: Building the output PDF — each redacted page is embedded as an image.

## Redaction Strategy

**Chosen approach: Rasterization with opaque overlays**

```
For each page that has redactions selected:
  1. Render original page to an offscreen <canvas> at 150 DPI
  2. Draw opaque black rectangles over every selected match bounding box
  3. Convert canvas to PNG blob via canvas.toBlob()
  4. Embed PNG as a full-page image in a new pdf-lib page
```

**Why this over other approaches:**

- **pdf-lib content-stream editing**: pdf-lib cannot selectively remove text operators from existing content streams — it can only append. The original text objects would remain in the PDF, recoverable with forensic tools. Unsafe for redaction.
- **pdf.js + custom content stream**: Possible in theory but requires deep PDF spec knowledge (text object encoding, font subsetting, content stream rewriting). Error-prone and out of scope.
- **Rasterization**: Guarantees all original content is destroyed. The output is a page-image PDF. Slight quality loss (raster, not vector), but for redaction this is the correct, safe trade-off.

**DPI choice: 150 DPI**
- 72 DPI → visibly blurry text in output
- 150 DPI → readable, modest file size increase (acceptable for a privacy tool)
- 300 DPI → very large files, unnecessary for redaction

## User Flow & UI

### Screen 1 — Upload
- Large drag-and-drop zone + file input button
- Shows thumbnail previews of uploaded files with page count
- "Continue" button (enabled when at least one file is loaded)
- Multi-file support: merge all uploaded PDFs into one redacted output

### Screen 2 — Search Terms
- Input field for adding terms/phrases (one at a time, Enter or button to add)
- Tags display for added terms, each with a remove (×) button
- "Find Matches" button triggers scan across all pages
- Suggested quick-add buttons for common PII: Name, Email, Phone, Address, SSN
- Loading state while scanning (show progress or spinner)

### Screen 3 — Preview & Select
- **Left panel (sidebar)**:
  - Match list grouped by page number
  - Each match shows: page number, term, surrounding context (snippet), checkbox (checked = will be redacted)
  - Toggle all on / Toggle all off buttons
  - Match count summary (e.g., "23 matches found · 18 selected")
- **Right panel (main)**:
  - `<canvas>` showing the current page
  - Selected matches highlighted with semi-transparent red overlays
  - Page navigation: Previous / Next + page number indicator
  - Zoom controls (fit-width preset; no free zoom to keep logic simple)
- **Navigation between matches**: clicking a match in the sidebar scrolls to and highlights that region on the canvas

### Screen 4 — Export
- Summary: file name, total pages, matches to redact
- "Download Redacted PDF" button
- Progress indicator during generation (per-page)
- Privacy notice: "Your file never left your browser"

## Text Extraction & Match Finding

### Extraction
```js
const textContent = await page.getTextContent();
// textContent.items: array of { str, transform: [scaleX, skewY, skewX, scaleY, translateX, translateY], width, height, fontName }
// Easiest reliable bounding box:
const tx = pdfjsLib.Util.transform(transform, [0, textContent.height]);
const [x, y] = [tx[4], tx[5]];
const bbox = { x, y, width: width, height: height };
```

**Note**: pdf.js uses a top-left origin for its internal text positioning. When rendering to canvas via `page.render()`, the viewport transformation handles the flip to screen coordinates. When extracting raw text items, we get position in PDF user-space coordinates (bottom-left origin). We use `page.getViewport()` to derive the conversion.

### Normalization
- All extracted text and search terms are lowercased before comparison
- Search terms are escaped for regex safety (user may type `.`, `*`, etc.)
- Whitespace normalization: collapse runs of whitespace in both source and terms before matching

### Multi-line phrase matching
- Extract text per page as an array of `{text, bbox}` spans (one per pdf.js text item)
- Concatenate spans into a full page string with index mapping: each character maps back to its source span index and offset
- For each search term, run a sliding-window scan over the concatenated string
- When a match is found, look up the span indices → collect all bounding boxes that cover the matched characters
- Merge overlapping/adjacent boxes into a single redaction rectangle

## Rendering Pipeline

```
page.render({ canvasContext, viewport })
  → canvas has the original page pixels

For each selected match bbox:
  draw red rectangle at (bbox.x, viewport.height - bbox.y - bbox.height)
  with width=bbox.width, height=bbox.height
  (flip Y because canvas is top-left origin, PDF coords are bottom-left)

canvas.toBlob('image/png') → PNG blob
pdf-lib: doc.addPage(pageWidth, pageHeight)
page.drawImage(pngEmbed, { x:0, y:0, width:pageWidth, height:pageHeight })
```

## Edge Cases & Handling

| Edge Case | Handling |
|---|---|
| PDF with no text layer (scanned, not OCR'd) | Show warning: "This PDF appears to have no searchable text. OCR must be performed before redaction." Disable search. |
| Very large PDF (100+ pages) | Render only current page in preview; show memory warning for export |
| Search term is regex special char (`.`, `*`, `+`) | Escape before building regex |
| Empty match list | "No matches found" message with suggestion to check spelling |
| Multi-file upload | Flatten all files sequentially; page numbers reset per file or show "File 1 – Page 3" |
| Browser doesn't support required APIs | Show error message with browser version info |
| User refreshes mid-process | All state lost (acceptable for a privacy tool — no server to restore from) |
| Text spans multiple pdf.js text items | Concatenation approach handles this (see Match Finding above) |
| Vertical or rotated text | pdf.js `transform` matrix provides correct bounds; rectangles drawn in page space handle any orientation |

## Dependencies (CDN — pinned versions)

```html
<!-- pdf-lib -->
<script src="https://cdnjs.cloudflare.com/ajax/libs/pdf-lib/1.17.1/pdf-lib.min.js"></script>

<!-- pdf.js + worker -->
<script type="module">
  import { pdfjsLib } from 'https://cdnjs.cloudflare.com/ajax/libs/pdf.js/4.4.168/pdf.min.mjs';
  pdfjsLib.GlobalWorkerOptions.workerSrc = 'https://cdnjs.cloudflare.com/ajax/libs/pdf.js/4.4.168/pdf.worker.min.mjs';
</script>
```

**CDN reliability note**: Both libraries are widely cached. For maximum offline robustness, a future improvement could bundle them locally, but CDN keeps the initial `index.html` simple and deployable.

## Privacy Verification Checklist

- [ ] No `fetch()`, `XMLHttpRequest`, or `navigator.sendBeacon()` calls

- [ ] All processing in main thread or Web Workers created from inline code
- [ ] README contains clear privacy statement

## Open Questions (to resolve before implementation)

1. **PDF.js version pin**: Should we pin a specific minor version to avoid CDN surprise updates? *(Recommendation: yes, pin to 4.4.168)*
2. **Single-page output quality**: Is 150 DPI acceptable, or should it be user-configurable? *(Recommendation: hardcode 150 DPI, mention in README)*
3. **Match snippet context**: How many characters of surrounding context to show? *(Recommendation: 50 chars before + 50 chars after)*
4. **Name of output file**: Use original filename + `-redacted.pdf`? *(Recommendation: yes, e.g., `contract-redacted.pdf`)*
